How to turn on VoiceOver

You will find VoiceOver under the ‘Accessibility’ menu in ‘Settings’. Here you can toggle VoiceOver on. Though you are likely to find it more convenient to turn VoiceOver on via an accessibility shortcut. This shortcut enables you to turn VoiceOver on from any screen or web page without going to the ‘Settings’ menu.

If you are just starting out, navigating using VoiceOver from within the ‘Settings’ menu to the component you are testing, could be a daunting process. Make sure VoiceOver is selected under ‘Accessibility Shortcut’ within the ‘Accessibility’ menu. On newer iPhones, you will now be able to triple click the side button to toggle VoiceOver on or off at any time. With older style iPhones the shortcut is a triple click of the ‘home’ button.

If at any time you would like to stop VoiceOver while announcing, do a two-finger tap.

It's speaking too fast

If you are new to VoiceOver, you may find the rate in which announcements are made too quick to understand. To find a speaking rate that you are comfortable with:

  • Select ‘Speaking rate’ in the Rotor menu by doing a two-finger rotation gesture (similar to how you might turn a dial with your thumb and finger) until you hear ‘Speaking rate’ announced
  • The speaking rate can now be adjusted by swiping up or down
  • Swipe down to decrease the speaking rate
  • Swipe up to increase the speaking rate

Testing in a foreign language

When testing with assistive technology it's important to test in a language that is supported by the assistive technology, and with a language that you can understand. For further information see the testing in a foreign language guide.

Before testing content in a foreign language, check what languages are supported by VoiceOver.

Step by step

We’ll take you step by step through how to test a component or feature. Following the same steps every time, ensures that everyone is testing using the same methods and using the most common techniques.

There are many VoiceOver gestures that you can use, though only a few are needed to perform the testing steps, all of which are explained below. The steps will enable anyone, in any role, to use VoiceOver for the first time. To become more familiar and proficient using AT, resist the temptation to cheat, always try to navigate like a user would.

As you go, make a note of any bugs you find.

Open Safari, this is the most used browser by VoiceOver users.

Step 4 - Get in position to start testing

Get in position to start testing by navigating as follows, depending on the context on which you are testing your component, either within a page or in isolation. This will ensure that when you start testing you don’t miss out any elements within the component that you can’t see, such as:

  • Semantics for any landmark or sectioning elements
  • Visually hidden off screen elements
Testing a component within a page

Navigate to the last element in the component before the one you're going to test.

  • Swipe right selects the next item, swipe right anywhere on the screen, until you reach the last element in the component before the component you're going to test. Ensure it’s a quick swipe, taking no more than a split second, otherwise what you have physically touched on the screen could be navigated to and read out instead.
  • You may find it quicker, if the component you are testing starts with a heading, to navigate to that heading via the ‘Rotor’ menu, then swipe left to select the previous item, until you hear the last element in the component before the one you're going to test being announced.
Testing a component in isolation

If the component is in isolation, ensure you are right at the top of the page.

  • To select the first item on the screen do a four-finger tap near the top of the screen
  • Swipe right selects the next item, swipe right though the address bar to the web page

Step 5 - Navigate through all items

You are now ready to start testing.

  • Swipe right selects the next item, you should enter the component.
  • Listen to what is announced and check that the semantics and announced content match the screen reader UX. If the documented screen reader UX is missing, ask UX for this before continuing.
  • If you need to hear what was announced again, swipe left to the previous item then swipe right again.
  • Swipe right through all items in the component, listening to what’s announced, until you reach the first item in the component after the component you are testing, or the bottom of the page if testing in isolation.

Navigating to the first item in the component after the one you're testing, will ensure that you don’t miss out any elements that you can’t see, such as a visually hidden off screen element.

Navigating through all items bug examples
  • You swipe right and nothing is announced
  • What's announced doesn’t match the screen reader UX
  • Semantics not being announced for any landmark or sectioning elements
  • Something is announced twice

Step 6 - Navigate by headings

  • Select ‘Headings’ in the Rotor menu by doing a two-finger rotation gesture (similar to how you might turn a dial with your thumb and finger) until you hear ‘Headings’ announced.
  • Headings are now selected as the method of navigation when swiping up or down.
  • Get in position to start testing, swipe up to navigate through headings until you reach the last heading in the component before the one you're testing, or if testing in isolation, to select the first item on the screen do a four-finger tap near the top of the screen.
  • Swipe down to read the first heading in the component you are testing. Check that the semantics and announced content match the screen reader UX.
  • If you need to hear what was announced again, swipe up to the previous heading then swipe down again.
  • Swipe down to go through all headings in the component, listening to what’s announced, until you reach the first heading in the component after the component you are testing, or the bottom of the page if testing in isolation.
Navigating by headings bug examples
  • You swipe down and nothing is announced
  • What's announced doesn’t match the screen reader UX
  • A heading was announced and none where defined in the screen reader UX
  • A heading is announced twice

  • Get in position to start testing, swipe up to navigate through links until you reach the last link in the component before the one you're testing, or if testing in isolation, to select the first item on the screen do a four-finger tap near the top of the screen.

To activate a link, once it has focus, double tap anywhere on the screen.

Step 9 - Accessibility acceptance criteria

Accessibility acceptance criteria can be used for additional manual testing steps specifc to your component and documented screen reader UX, specific VoiceOver gestures maybe needed.

  • Look at the accessibility acceptance criteria. If the accessibility acceptance criteria is missing, ask your team's business analyst for the criteria before continuing.
  • Check that any criteria that is specific to using a screen reader is met with VoiceOver iOS.
